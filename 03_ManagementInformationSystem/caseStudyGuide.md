# AX/DX 창업·MIS 사례개발 실습 가이드
## 목적
본 과제는 기업을 설명하는 보고서를 작성하는 것이 아니라,
**비즈니스 모델과 데이터 프로세스가 언제 성공하고 언제 실패하는지(경계조건)를 구조화**하여
재현 가능한 사례 데이터셋을 구축하는 것을 목표로 한다.

즉, 사례 분석이 아니라 "실험 가능한 케이스 레코드(case record)"를 만든다.

---

## 팀 구성 및 비교 방식
각 팀은 다음과 같은 비교 구조를 따른다.

- 국내/동문 스타트업 1개
- 글로벌 또는 확장 성공 기업 1개

비교 목적:
> 동일 문제를 서로 다른 규모/환경에서 어떻게 해결했고 어디서 실패했는가

---

## 제출물 구성
총 제출물은 4개 파트로 구성된다.

1) Case Metadata
2) System & Data Flow
3) Success vs Failure Boundary
4) Intervention & Improvement Proposal

---

# Part 1. Case Metadata (필수 구조화 정보)
다음 표를 채워라.

| 항목 | 내용 |
|---|---|
| Case ID | 학기-그룹-팀번호 |
| 산업 도메인 | 제조 / 바이오 / SaaS / 플랫폼 등 |
| 기업 유형 | 스타트업 / 스케일업 / 대기업 |
| 핵심 문제 | 기업이 해결하려는 의사결정 문제 1문장 |
| 사용자/이해관계자 | 고객, 운영자, 파트너, 규제자 등 |
| 수익 모델 | 구독 / 수수료 / 장비판매 / 광고 등 |

# Part 2. System & MIS 관점 분석
## 2-1 Data Flow Diagram 작성
다음 요소를 반드시 포함하여 그린다.

- 데이터 입력원 (센서, 사용자행동, 거래, 콘텐츠 등)
- 저장 위치 (DB, 로그, 클라우드, LIMS 등)
- 처리 방식 (AI, 규칙기반, 수작업)
- 의사결정 지점 (자동/사람)
- 결과 출력 (추천, 제어, 승인, 알림)

## 2-2 핵심 데이터 자산 정의
| 데이터 종류 | 수집 목적 | 의사결정 영향 |
|---|---|---|

# Part 3. Success vs Failure Boundary 분석 (핵심 파트)
기업을 소개하지 말고 반드시 "언제 깨지는가"를 찾아라.

## 3-1 성공 조건
시스템이 정상 작동하는 환경 조건을 정의

| 변수 | 범위 | 결과 |
|---|---|---|

## 3-2 실패 사건 정의
다음 문장 형식으로 작성:

> "○○ 조건에서 △△가 발생하며 시스템이 목표를 달성하지 못했다."

| 실패 유형 | 원인 분류 | 영향 |
|---|---|---|

원인 분류 택1:
- D: 데이터 문제
- M: 모델/알고리즘
- O: 운영/시스템
- E: 경제/인센티브
- S: 사용자/사회 수용성

## 3-3 Boundary Condition 정리
| 조건 변수 | 임계값 | 결과 변화 |
|---|---|---|

# 핵심 규칙 (가장 중요)
이 과제의 목표는 성공사례를 설명하는 것이 아니다.

> 성공 이유 1개보다 실패 조건 1개가 더 높은 점수를 받는다.

우리는 기업을 평가하는 것이 아니라
시스템이 작동하지 않는 조건을 찾아
재현 가능한 지식으로 만드는 것이 목표다.
